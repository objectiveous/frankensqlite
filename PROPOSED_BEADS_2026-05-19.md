# Proposed Beads: Multi-Process Durability + File-Backed Time Travel

> Drafted 2026-05-19 from a GitHub-issue triage pass. The proposed
> beads cover three open issues and one observed in-the-wild corruption
> pattern. They are written as a plan, not yet imported, because the
> live beads DB at `.beads/beads.db` is itself page-corrupted (see
> Epic 0) and `br create` cannot write to it.

## Source issues

- **frankensqlite#70** — "Multi-process concurrent access consistently corrupts or stales the database — success criterion: robust behavior under swarm write load." Open, P0, filed 2026-05-18.
- **frankensqlite#82** — "Feature: Add File-Backed Native-Mode Time Travel for `FOR SYSTEM_TIME AS OF`." Open, filed 2026-05-18. Depends on the durable history substrate that was deferred when #30 was closed.
- **mcp_agent_mail_rust#118** — "MCP `send_message` panics serve-http with `RefCell already borrowed`". The panic site is reported at `crates/fsqlite-core/src/connection.rs:47590` — upstream from `mcp_agent_mail_rust`. Frankensqlite needs the actual fix.
- **In-vivo evidence** — the beads DB in *this* repo is currently corrupted with 33+ multi-reference page errors (`Tree 20 page 20 cell N: 2nd reference to page M`), with 28 preserved `.rebuild-failed` artifacts in `.beads/.br_recovery/` going back to 2026-03-19. This is exactly the failure mode #70 describes — page double-references that survive checkpoints — so a deliverable here is to capture forensic state from that DB before any rebuild.

### bd-NEW-1.4 — Two-phase commit fence between WAL frame append and page table publish

```bash
br create --type=task --priority=0 --title="Two-phase commit: WAL fsync MUST happen before CommitIndex publish" \
  --description='A common failure mode for multi-process MVCC is: writer A appends WAL frames, flips CommitIndex[pgno]=N to publish, then crashes before fsync. Subsequent reader B sees CommitIndex=N, looks up the frame, finds it half-written or absent → torn read. The fix: enforce a strict two-phase ordering — (1) write WAL frames + fsync(wal_fd), (2) memory-barrier (Release), (3) update CommitIndex with Release ordering. Audit every CommitIndex publish site to confirm it is preceded by an explicit fdatasync (or fdatasync-equivalent on platforms without it) and a Release barrier. Add a `debug_assertions`-only invariant check that records the last fsync sequence per page and panics if Publish happens before Fsync. Acceptance: cargo asan + cargo loom both green on tests/wal_commit_ordering.rs.'
```

## Epic 2 — File-backed native-mode time travel (#82)

`FOR SYSTEM_TIME AS OF COMMITSEQ N` works for `:memory:` / compatibility runtime via the in-memory snapshot ring (#23). On a file-backed DB the query fails with `time-travel: no snapshot available for COMMITSEQ N`. The path to fix this overlaps with the durable native commit/ref/history substrate that was tracked in #30 (closed deferred).

Decide whether to: option A — revive #30 in full (months of work, broader payoff), option B — land a minimal commit-snapshot log (weeks, scoped to #82).
